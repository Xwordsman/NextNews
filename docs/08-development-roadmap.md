# 开发顺序

NextNews 的开发顺序应该围绕“先形成数据闭环，再扩展运营能力”。不要先做一堆静态页面，也不要一开始铺满所有后台菜单。第一版最重要的是让频道可以被后台管理、被 Worker 抓取、被写入数据库、被前台读取。

## 总体顺序

```txt
0. 规划确认
1. 项目骨架
2. 数据库与迁移
3. 认证与后台壳
4. 内容源基础管理
5. 频道定义机制
6. Worker 抓取闭环
7. 前台首页与频道页
8. 个人中心与订阅
9. Docker 部署
10. 历史、日报、榜中榜、话题、追踪
```

## 当前实现进度

截至当前版本，阶段 1-9 已完成第一版闭环，阶段 10 已完成运营增强和商业化控制的基础闭环：

- 后台屏蔽内容：最新内容一键屏蔽、手动添加屏蔽 URL、取消屏蔽。
- 前台屏蔽过滤：首页、站点、分类、频道、快照、日报、话题、动态和追踪统一过滤。
- 首页模块配置：后台可以控制首页模块标题、排序、启停和展示数量。
- 日报配置：后台生成/发布/下线日报，前台 `/daily` 和 `/daily/[reportDate]` 支持按日期查看日报。
- 榜中榜配置：后台创建榜中榜配置并绑定频道，前台 `/rankings` 聚合多频道快照。
- 话题配置：后台创建话题、关键词、启停、前台显示和人工挂载内容，前台 `/topics/[slug]` 支持详情页。
- 追踪规则：用户在 `/tracking` 创建/删除关键词规则，命中结果落库，后台可以查看规则和命中。
- 站内通知：追踪命中可生成通知，前台 `/notifications` 支持查看和标记已读，后台可查看通知。
- 历史归档：频道历史页 `/channels/[siteSlug]/[channelSlug]/history` 支持年/月筛选快照。
- 站内搜索：前台 `/search` 支持检索已入库快照内容、站点和频道。
- 采集闭环：频道页和采集任务页可手动采集，失败记录可发起 retry。
- 部署闭环：首次安装自动迁移，已安装环境拉新镜像后自动补齐最新迁移。

上一轮已完成：会员历史权限、搜索筛选与高亮、订阅频道通知策略、运营数据统计和更细粒度的日报人工编辑。

本轮继续完成：商业化总开关、会员套餐与订单预留、搜索日志、收藏、阅读历史、日报模板、异步任务队列预留、后台操作日志、运营统计导出和前台 `/go` 跳转记录。

## 阶段 0：规划确认

当前文档阶段已经完成主要方向确认：

- 产品定位：新闻与热榜聚合系统。
- 数据库：PostgreSQL。
- 后台路径：`/admin`。
- 部署方式：服务器 Docker Compose，适配宝塔面板。
- 容器形态：`web`、`worker`、`postgres`、`redis`。
- 频道接入方式：代码仓库中的频道定义文件 + 后台运营配置。

进入开发前还需要在实现层面确认：

- 认证方案选择。
- UI 组件方案选择。
- 包管理器和代码格式化约定。

已确认的实现决策：

- ORM/数据库工具：Drizzle ORM + drizzle-kit + postgres。

其他选项属于实现决策，不影响当前产品架构。

## 阶段 1：项目骨架

目标：让项目能稳定启动，并形成后续所有模块的目录边界。

交付：

- 创建 Next.js 16.2.9 项目。
- 配置 TypeScript、Lint、格式化、路径别名。
- 建立 `src/app`、`src/features`、`src/server`、`src/components`、`src/lib`、`src/types`。
- 建立 `.env.example`。
- 建立基础布局、主题、错误页、加载态。
- 把 `ui` 静态稿拆解为后续 React 组件参考，不直接复制静态脚本。

验收：

- 本地开发服务可启动。
- 首页和 `/admin` 可以打开占位页面。
- 目录结构符合文档约定。

## 阶段 2：数据库与迁移

目标：先把核心数据模型立住，避免后面页面和后台无处落数据。

交付：

- PostgreSQL 连接。
- 数据库 schema 和迁移脚本。
- 核心表：
  - `sys_user`
  - `biz_site`
  - `biz_category`
  - `biz_channel`
  - `rel_channel_category`
  - `log_crawl_run`
  - `biz_channel_snapshot`
  - `biz_snapshot_item`
  - `biz_content_item`
- 初始化能力：
  - 首次访问 `/install` 自动进入安装向导。
  - 安装向导创建第一个管理员。
  - 安装向导自动执行数据库迁移。
  - 安装向导写入基础分类、站点和频道。

验收：

- 首次访问可以通过安装向导完成迁移和初始化。
- 数据库中能看到站点、分类、频道和管理员账号。

## 阶段 3：认证与后台壳

目标：先把后台入口保护起来，再逐步填功能页面。

交付：

- 管理员登录。
- `/admin` 路由保护。
- 后台基础布局：侧边栏、顶部栏、面包屑、退出登录。
- 工作台基础指标占位。
- 操作反馈组件：确认、提示、错误展示。

验收：

- 未登录不能访问 `/admin`。
- 管理员可以登录和退出。
- 后台菜单结构和文档一致。

## 阶段 4：内容源基础管理

目标：完成后台最核心的基础资料 CRUD。

交付：

- 站点管理。
- 分类管理。
- 频道管理。
- 频道和分类绑定。
- 频道运营字段：
  - 是否启用抓取。
  - 抓取间隔。
  - 是否公开。
  - 是否首页显示。
  - 是否允许订阅。
  - 排序和权重。

验收：

- 可以在后台创建站点、分类、频道。
- 频道必须绑定 `definition_key`。
- 后台不允许填写任意代码、动态 import 路径、CSS selector、JSON path。

## 阶段 5：频道定义机制

目标：建立 RSS 和复杂爬虫统一入口。

交付：

- `src/server/channels/registry.ts`。
- `defineChannel` 辅助函数。
- 统一 `NewsItem` 输出结构。
- 第一个 RSS 示例频道。
- 第一个自定义 adapter 示例频道。
- 后台可读取注册表中的可用频道定义。

验收：

- `definition_key` 只能从注册表匹配。
- 注册表中不存在的频道不能执行抓取。
- 新增频道定义文件后，后台可以选择并绑定。

## 阶段 6：Worker 抓取闭环

目标：形成最小可运行的数据闭环。

交付：

- `pnpm worker` 常驻入口。
- 调度器扫描需要抓取的频道。
- PostgreSQL advisory lock，避免同频道并发抓取。
- `CRAWL_CONCURRENCY` 控制 Worker 并发数量。
- `CRAWL_RUNNING_TIMEOUT_SECONDS` 清理卡住的 running 任务。
- 抓取运行日志。
- 快照写入。
- 快照条目写入。
- 内容主表去重。
- 更新频道 `last_snapshot_id`、`last_crawl_at`、`last_success_at`。
- 后台手动触发抓取。

验收：

- 启动 Worker 后，启用的频道会按间隔抓取。
- 后台可以看到抓取日志。
- 前台或数据库可以读取最新快照。
- 抓取失败会写入失败原因，不会拖垮 Web 服务。

## 阶段 7：前台首页与频道页

目标：把已有 `ui` 设计稿转化为真实数据页面。

交付：

- 首页读取后台配置和最新快照。
- 热门站点入口。
- 分类/频道导航。
- 热榜卡片。
- 频道详情页。
- 频道历史入口占位。
- 搜索框基础 UI。
- 移动端适配。

验收：

- 首页不硬编码频道列表。
- 频道卡片读取数据库最新快照。
- 后台关闭首页显示后，前台对应频道不展示。
- 页面视觉参考 `ui` 静态稿，并结合 TopHub 的信息密度。

## 阶段 8：个人中心与订阅

目标：让前台用户具备最基础的个性化能力。

交付：

- 前台用户注册/登录。
- 个人中心。
- 频道订阅和取消订阅。
- 订阅动态。
- 后台查看用户和订阅关系。

验收：

- 用户可以订阅允许订阅的频道。
- 不允许订阅的频道不显示订阅入口。
- 订阅动态只显示用户订阅范围内的内容。

## 阶段 9：Docker 部署

目标：让项目能按宝塔容器编排方式部署。

交付：

- `Dockerfile`。
- `docker-compose.yml`。
- `.env.example`。
- Web 容器启动命令。
- Worker 容器启动命令。
- PostgreSQL volume。
- Redis volume，为缓存、队列和后续分布式调度预留。
- 健康检查。
- 首次访问安装向导。
- Worker 未安装时等待初始化，不因为表不存在反复崩溃。

验收：

- 宝塔 Docker 编排启动后有 4 个容器。
- 只有 Web 对公网暴露端口。
- Worker 不暴露公网端口。
- 重启后 PostgreSQL 数据不丢失。
- 新数据库首次访问会跳转 `/install`，无需手动进入容器执行迁移。

## 阶段 10：运营增强能力

这些能力不阻塞基础 MVP，但会决定产品后续能否形成运营和商业化能力。

优先顺序建议：

1. 历史快照查询。
2. 首页模块配置增强。
3. 内容屏蔽。
4. 日报。
5. 榜中榜。
6. 话题聚合。
7. 追踪规则。
8. 会员历史权限。
9. 搜索服务。

验收：

- 每个能力都基于已入库快照，不直接实时请求第三方站点。
- 后台可以控制展示、隐藏、排序和发布状态。
- 大表查询必须带频道和时间范围条件。

当前已完成阶段 10 的第一轮增强：

- 日报已支持按日期查看历史详情。
- 话题已支持人工关联条目和详情页。
- 追踪已从实时查询升级为命中记录落库，并可生成站内通知。
- 榜中榜已从首页频道排序升级为跨频道热度聚合。
- 频道历史已支持年/月筛选。
- 首页模块已支持后台配置展示数量、排序和启停。
- 搜索已支持基于入库快照的站内检索。

上一轮补齐的能力：

- 会员历史权限和套餐边界：后台可为用户配置 `history_days`，前台历史快照和日报详情按权益过滤。
- 搜索筛选和高亮：前台 `/search` 支持关键词、站点、频道、日期范围，并高亮命中词。
- 订阅频道通知策略：订阅用户可开启频道更新通知，Worker 写入新快照后生成站内通知。
- 运营数据统计：后台 `/admin/operations/analytics` 汇总频道、快照、失败任务、用户、订阅、会员和通知指标。
- 日报人工选题和内容编排：后台可把快照条目加入日报精选，前台日报详情优先展示人工精选。

本轮继续补齐的能力：

- 商业化总开关：`commerce.enabled` 默认关闭，后台可控制前台是否展示会员购买入口。
- 会员套餐与订单预留：后台维护 `membership_plan`，前台 `/membership` 在开启商业化后创建 `pending` 订单。
- 搜索日志：`search.logging_enabled` 控制是否记录搜索关键词、筛选条件和结果数。
- 收藏与阅读历史：前台提供 `/bookmarks`、`/history`，主要外链统一经 `/go/[snapshotItemId]` 记录阅读。
- 日报模板：后台维护 `biz_daily_report_template`，为后续自动生成和审核流预留。
- 异步任务队列：`sys_job_queue` 和 `jobs.async_enabled` 为 Worker 消费派生任务预留。
- 操作日志与导出：后台关键操作写入 `sys_operation_log`，运营统计支持受控 CSV 导出。

下一步重点应从“首版运营控制”走向“更深的自动化与支付闭环”：

- 支付网关、回调验签、订单状态变更和会员自动授权。
- Worker 消费 `sys_job_queue`，处理日报生成、通知聚合和异常预警。
- 搜索索引优化、热门搜索词趋势和更强的搜索质量分析。
- 通知策略继续细化，例如按排名变化、热度变化和用户偏好触发。
- 运营统计趋势图、异常预警和更多可导出的运营报表。
- 频道源扩展放到系统主流程稳定之后再集中处理。

## 第一版 MVP 边界

第一版应该做到：

- 后台能管理站点、分类、频道。
- 频道能绑定代码中的频道定义。
- Worker 能抓取并入库。
- 前台首页能展示后台配置的频道。
- 频道页能展示最新快照。
- 用户能订阅频道并查看订阅动态。
- Docker Compose 能在服务器上运行。

第一版不强求：

- 完整会员系统。
- 完整搜索服务。
- 自动话题聚类。
- 很复杂的榜中榜算法。
- 多管理员 RBAC。
- Vercel 或 Cloudflare Pages 部署。

## 不建议提前做的事

- 不要先做完整视觉页面，再回头补数据结构。
- 不要先铺满所有后台菜单，再逐个接假数据。
- 不要把抓取任务写进页面请求。
- 不要在后台提供在线写 JS 爬虫能力。
- 不要为了早期省事跳过历史快照表。
- 不要一开始就引入复杂微服务。

## 开发检查点

每完成一个阶段，都要检查：

- 是否有真实数据闭环。
- 是否符合后台控制前台的原则。
- 是否保留历史快照。
- 是否避免把采集代码暴露给后台在线编辑。
- 是否能在 Docker 环境运行。
- 是否给下一阶段留下清晰接口。
